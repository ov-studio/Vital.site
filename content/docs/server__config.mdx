## Overview

Every **Vital.server** ships with a `config.yaml` in its root directory, pre-populated with sensible defaults. It is the single source of truth for runtime behaviour вЂ” modify the values, restart the server, and changes take effect immediately.

If the config file is absent or malformed, the server falls back to built-in defaults and emits a diagnostic to the console. A missing file is non-fatal.

<Callout type="info">
  **Where the config lives**

  - **Fixed location** вЂ” `config.yaml` must sit at the root of your `Vital.server/` directory.
  - **Single file** вЂ” all sections (`server`, `network`, `http`, `bootstrap`, `social`) live in this one file.
  - **Restart to apply** вЂ” changes are not hot-reloaded; a server restart is required for any edit to take effect.
</Callout>

## Configuration

Default configuration as shipped with every Vital.server.

```yaml title="Vital.server/config.yaml"
# =============================================================================
# Vital Sandbox вЂ” Server Configuration
# =============================================================================

server:
  name: "Vital.sandbox Server"
  version: "1.0.0"
  description: "A Vital.sandbox server"

network:
  port: 7777
  max_peers: 32

http:
  port: 7778

bootstrap:
  #- "runcode"

social:
  discord: ""
  website: ""
```

<Callout type="warn">
  **All keys are optional.**

  - Every key has a built-in default and can be omitted without error.
  - An absent or malformed `config.yaml` is non-fatal вЂ” the server starts with defaults and logs a warning.
</Callout>

#### Server

Controls the display name, version string, and description exposed to clients and integrations.

```yaml
server:
  name: "Vital.sandbox Server"
  version: "1.0.0"
  description: "A Vital.sandbox server"
```

| Key | Type | Default | Description |
|---|---|---|---|
| `name` | `string` | `"Vital.sandbox Server"` | Display name shown in the master-list |
| `version` | `string` | `"1.0.0"` | Free-text version identifier - not validated or compared |
| `description` | `string` | `""` | Optional description shown in the master-list |

---

#### Network

Controls the ENet multiplayer transport layer. Clients connect to the port defined here.

```yaml
network:
  port: 7777
  max_peers: 32
```

<Callout type="warn">
  **Both values directly affect client connectivity вЂ” change them with care.**

  - **`port`** вЂ” clients and any firewall or port-forwarding rules must be updated to match.
  - **`max_peers`** вЂ” once the limit is reached, incoming connections are refused until a slot becomes available.
</Callout>

| Key | Type | Default | Description |
|---|---|---|---|
| `port` | `integer` | `7777` | UDP/TCP port the ENet multiplayer server listens on |
| `max_peers` | `integer` | `32` | Maximum number of concurrent client connections |

---

#### HTTP

Controls the asset delivery server. Clients fetch assets such as models and textures from this port at runtime.

```yaml
http:
  port: 7778
```

<Callout type="info">
  **The HTTP server operates independently of the ENet transport layer.**

  - **Both ports must be reachable** вЂ” blocking either one will result in a failed or degraded session.
  - **Separate firewall rules** вЂ” ensure both ports are open and forwarded correctly in your network configuration.
</Callout>

| Key | Type | Default | Description |
|---|---|---|---|
| `port` | `integer` | `7778` | Port the HTTP asset delivery server listens on |

---

#### Bootstrap

Defines resources that are automatically started after the initial resource scan on server boot. Entries are processed in order and respect full dependency resolution вЂ” if a bootstrap resource depends on another resource, that dependency is started first.

```yaml
bootstrap:
  - "runcode"
  - "gamemode"
```

<Callout type="info">
  **Bootstrap runs once at startup, after the resource scan completes.**

  - **Order is preserved** вЂ” resources start top-to-bottom as listed.
  - **Dependencies are resolved automatically** вЂ” a resource's deps start before it, even if not listed here.
  - **Each entry must be a valid resource name** вЂ” must match a folder under `resources/` (`a-z`, `0-9`, `_` only).
  - **Already-running resources are skipped** вЂ” no error is raised if a resource is already running.
</Callout>

| Value | Type | Description |
|---|---|---|
| entry | `string` | Name of a resource folder under `resources/` to start at boot |

---

#### Social

Defines community links surfaced in the master-list.

```yaml
social:
  discord: ""
  website: ""
```

| Key | Type | Default | Description |
|---|---|---|---|
| `discord` | `string` | `""` | Discord server invite URL shown in the master-list |
| `website` | `string` | `""` | Community or project website URL shown in the master-list |
