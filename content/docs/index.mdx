## Overview

**Vital.sandbox** is the next-gen game sandbox by [ov-studio](https://ko-fi.com/ovstudio) — **fully open-source**, battle-tested, built on the **Godot Engine**, and engineered from the ground up in **C++17** for maximum performance and control.

One language. Infinite power. Rendering, networking, threading, asset delivery — the entire game, top to bottom, all from Lua.

Four principles drive everything under the hood:

- **⚡Performance-first** — Lean by design. No bloat, no unnecessary abstractions. Nothing stands between your scripts and the engine.

- **🧱 Modular by nature** — Strict C++17 module system. Use what you need, extend what you want, leave the rest untouched.

- **🔧 Built for developers** — Async/await, promises, heartbeats, event-driven patterns — modern concurrency that feels natural without sacrificing low-level control.

- **🌍 Open to the core** — No licensing fees, no royalties, no strings attached. What you build belongs to you — completely and unconditionally.

## Features

Vital.sandbox ships with an integrated suite of tools covering every layer of application development — from low-level threading to asset delivery and third-party integrations.

### Core

| Tool | Description |
|---|---|
| 🔐 **Crypto** | Cryptographic primitives for securing sensitive data and payloads |
| 🗜️ **Shrinker** | Asset and data compression to reduce memory footprint and transfer overhead |
| 📚 **Stack** | Managed stack abstraction for structured, predictable data handling |
| 🔍 **Inspect** | Hardware serial inspection and device fingerprint generation |
| 🖥️ **Console** | Built-in runtime console for executing commands and viewing debug, info, and error output |

### Sandboxing

| Property | Details |
|---|---|
| 🗣️ **Language** | Lua |
| 🔒 **Isolation** | Full — no access to host internals or native state |
| 🧩 **Use Cases** | User scripting, modding, plugin systems |

### Networking

| Tool | Description |
|---|---|
| 🌍 **Rest** | HTTP/REST client for communicating with external APIs |
| 📡 **Robust Networking** | High-performance transport layer with reliability and delivery guarantees |
| ⏱️ **Scheduling** | Precise task scheduling for time-sensitive execution control |

### Threading

| Primitive | Description |
|---|---|
| ⚡ **Async / Await** | First-class asynchronous execution with clean, readable control flow |
| 🔗 **Promises** | Deferred values with full chaining and resolution support |
| 💓 **Heartbeats** | Scheduled recurring callbacks at defined intervals |
| 🧶 **Threader** | Low-level thread management, pooling, and lifecycle control |

### Rendering

Full programmatic control over every rendering primitive at runtime, with no overhead.

| API | Description |
|---|---|
| 🖌️ **Canvas** | 2D drawing surface with support for shapes, images, and text composited each frame |
| 🎯 **RenderTarget** | Off-screen render surfaces — create, bind, and sample as textures at runtime |
| 🖼️ **Texture** | Runtime texture loading, unloading, and sampler control |
| 🔤 **Font** | Font loading and text metrics for canvas-level text rendering |
| 🌐 **Webview** | Embedded web content rendering within the application window |

### Models

Model API covers the full runtime lifecycle and is designed with customization systems in mind.

| Feature | Description |
|---|---|
| 📦 **Loading / Unloading** | Load and unload named model assets at runtime without restart |
| 🧬 **Instantiation** | Create and destroy independent runtime instances from a shared loaded asset |
| 📐 **Transform** | Full 3D transformation control per instance — position, rotation, and scale |
| 🎬 **Animation** | Per-instance animation playback with speed control, looping, pause, and resume |
| 🎭 **Blend Shapes** | Per-component blend shape value control — facial expressions, body morphs, and more |
| 👗 **Component Visibility** | Per-component mesh visibility control — built for clothing and equipment systems |

### Integrations

| Integration | Description |
|---|---|
| 🎮 **Discord SDK** | Native Rich Presence and Discord API integration |
| 📣 **Event Tool** | Pub/sub event system for decoupled inter-module communication |
