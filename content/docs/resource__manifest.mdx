## Overview

Every resource in **Vital.sandbox** lives in its own folder under the server's `resources/` directory, described by a single `manifest.yaml` at its root. The Resource Manager scans this directory on server startup, parses each manifest, and tracks the resulting metadata for the lifetime of the server.

**Server builds** read `manifest.yaml` directly from disk, validate it, resolve dependencies, and broadcast the resulting metadata to connected clients.

**Client builds** never read `manifest.yaml` — they receive an equivalent, pre-resolved manifest over the network from the server and register it locally.

<Callout type="info">
  **Where resources live**

  - **Folder = resource** — each resource is a folder named after it, e.g. `resources/inventory/`.
  - **Fixed filename** — the manifest must be named exactly `manifest.yaml` and sit at the folder's root.
  - **Folder name is the reference** — it becomes the resource's `ref` and is used everywhere the resource is addressed.
</Callout>

```txt title="Example resource folder"
resources/
└── inventory/
    ├── manifest.yaml
    ├── client.lua
    ├── client2.lua
    └── assets/
        └── ...
```

Script and file paths declared in the manifest are resolved relative to this folder.

## Resource

#### Naming conventions:

- must not be empty
- must pass filesystem path sanitization (no path traversal or unsafe path characters)
- must contain only alphanumeric characters or underscores (`_`) — no spaces, dots, or slashes

<Callout type="idea">
  **Restarting resource**

  - Restarting reloads the manifest, then compares old vs. new script hashes, file hashes, and dependency lists, reporting exactly what was added, modified, or deleted before stopping and re-starting the resource.
  - This makes it possible to see at a glance whether a restart picked up the change you expected.
</Callout>

<Callout type="info">
  **Reconnecting / late-joining clients**

  - On peer join, the server broadcasts the full asset manifest, then sends a `resource:started` packet for every currently running resource, in the order they were originally started.
  - This lets a client that connects mid-session receive every active resource without a full server restart.
</Callout>

## Manifest

Full example showing every supported top-level key:

```yaml title="resources/<name>/manifest.yaml"
name: "My Resource"
author: "YourName"
version: "1.0.0"

author_url: "https://github.com/YourName" # Used only by vault
banner: "https://raw.githubusercontent.com/<org>/<repo>/main/resources/<name>/banner.png" # Used only by vault
tagline: "One-line summary of what this resource does" # Used only by vault
featured: false # Used only by vault

description: |  # Used only by vault
  Longer description shown in the Vault modal.
  Supports multiple lines like this.

tags:  # Used only by vault
  - "utility"

dependencies:
  - "inventory"

scripts:
  - { src: "client.lua", type: "client" }
  - { src: "client2.lua", type: "client" }
  - { src: "shared/*.lua", type: "shared" }
  - { src: "server.lua", type: "server" }

files:
  - "assets/**"

```

<Callout type="warn">
  **Only `scripts` is mandatory.**

  - A manifest with no `scripts` key, or where `scripts` isn't a list, is rejected outright and the resource is skipped during scan.
  - `name`, `author`, `version`, `files`, and `dependencies` are all optional.
</Callout>

| Key | Type | Required | Default | Used in | Description |
|---|---|---|---|---|---|
| `name` | `string` | No | `ref` | Sandbox, Vault | Display name for the resource<br/>Falls back to resource's reference if omitted |
| `author` | `string` | No | `""` | Sandbox, Vault | Free-text author/credit string - not validated |
| `version` | `string` | No | `""` | Sandbox, Vault | Free-text version string - not validated |
| `scripts` | `string[]` | Yes | None | Sandbox | List of script entries to compile and execute for this resource |
| `files` | `string[]` | No | `[]` | Sandbox | Glob patterns for non-script assets (models, textures, sounds, etc) |
| `dependencies` | `string[]` | No | `[]` | Sandbox | Names of other resources that must be running before this one starts |
| `author_url` | `string` | No | `https://github.com/<author>` | Vault | Link wrapped around the author name in the Vault modal |
| `tagline` | `string` | No | `"N/A"` | Vault | Short one-line summary shown on the Vault card + modal |
| `description` | `string` | No | `"N/A"` | Vault | Longer text shown in the Vault modal<br/>Use a YAML block scalar (`\|`) for line breaks - `<br/>` renders as plain text, not HTML |
| `tags` | `string[]` | No | `[]` | Vault | Filters shown on the Vault card - only known tags are kept: `gamemode`, `utility`, `ui`, `physics`, `audio`, `networking`, `tools` |
| `banner` | `string` | No | `null` | Vault | Full URL to a banner image - shows a placeholder icon if omitted<br/>No path resolution yet, so this must be a full URL, not a relative filename |
| `featured` | `boolean` | No | `false` | Vault | Pins the resource to the top of the Vault grid |

#### Scripts

Each entry in `scripts` is a map with two required fields:

```yaml
scripts:
  - { src: "client.lua", type: "client" }
```

<Callout type="warn">
  **Invalid or unmatched entries fail the whole manifest.**

  - A script entry missing `src` or `type` is a hard error — the manifest fails to parse.
  - `type` must be exactly `"shared"`, `"server"`, or `"client"` — anything else is rejected.
  - A `src` pattern that matches zero files is rejected — there's no "optional" script.
</Callout>

| Field | Type | Description |
|---|---|---|
| `src` | `string` | Path or glob pattern to a `.lua` script, relative to the resource folder<br/>Expanded against the filesystem at load time; a pattern that matches nothing is an error |
| `type` | `string` | One of the below script types:<br/>• `"shared"` - Loaded and executed on server and client both<br/>• `"server"` - Loaded and executed on server only<br/>• `"client"` - Loaded and executed on client only |

---

#### Files

A flat list of glob patterns for non-script assets — models, textures, sounds, config, anything the resource needs to ship.<br/>
Like scripts, each pattern is expanded on disk; a pattern matching nothing produces an error and the whole manifest fails.

<Callout type="info">
  **Files are hashed at parse time**.
  
  - Changes after every restart can be detected as per [Resource](#resource) section.
  - Models declared under `files` are additionally auto-detected and loaded as part of resource startup.
</Callout>

```yaml
files:
  - "assets/**"
  - "models/*.glb"
  - "config.json"
```

---

#### Dependencies

A list of other resource names that must be running before this resource starts.<br/>
Resource Manager resolves the full dependency graph recursively and starts dependencies in the correct order before starting the resource itself.

| Rule | Behaviour |
|---|---|
| `invalid-name` | Any dependency name failing the [naming conventions](#naming-conventions) is rejected with an error |
| `self-dependency` | A resource cannot list itself as a dependency - rejected outright |
| `missing dependency` | If a listed dependency isn't a known resource, resolution fails and the resource does not start |
| `circular-dependency` | Detected via a recursion stack; a cycle (e.g. `A → B → A`) aborts the start with the full chain reported |
| `already-running` | Dependencies already running are skipped - they are not restarted |
